03 内存泄漏分析 ·「批量制造内存泄漏」

MEM-1

模块级引用无限累积 + 游离 DOM 节点

Memory Leak

触发控件：button「批量制造内存泄漏」 · 每次点击追加 10,000 条保留引用

实测堆内存（1x CPU · 堆上限 4192 MB · performance.memory）
状态leakedRecordsusedJSHeapSize
初始01.4 MB
点击 1 次10,000194.5 MB
点击 3 次30,000577.7 MB
继续操作—渲染进程 CRASHED 🧨
整页 reload 后01.7 MB

线性增长 ≈ 192 MB / 次点击，约 19.7 KB / 条记录。第 3 次后在快照 / 重置时 tab 直接 Target crashed。

三个叠加的泄漏因素
① 保留引用
模块级数组永不释放

leakedRecords 只 push 不清空，GC 视其始终可达。

② 游离 DOM
detached <section> 节点

createElement 后从不插入文档，却被数组强引用。每次点击 10,000 个。

③ 大 payload
每条 5,000 槽数组

new Array(5_000).fill(...) 单条 ≈40KB 背板，一次点击约 400MB 级分配。

根因定位（仅记录，未修改）
// public/app.js:27-28 —— 模块作用域，页面存活期间永不释放
const leakBatchSize = 10_000;
const leakedRecords = [];

// public/app.js:106-121
leakMemory.addEventListener('click', () => {
  for (let index = 0; index < leakBatchSize; index += 1) {
    const detachedNode = document.createElement('section');  // ② 从不 append 到文档
    detachedNode.textContent = `leaked node ${leakId}`;
    leakedRecords.push({                                     // ① 强引用，永不清空
      createdAt: new Date().toISOString(),
      detachedNode,
      payload: new Array(5_000).fill(`order-leak-${leakId}`), // ③ ≈40KB / 条
    });
  }
});
没有任何释放路径

「重置演示」(app.js:123-128) 只重置 orders / pendingOnly / feedback，完全没碰 leakedRecords——没有 .length = 0、没有 removeEventListener。点过之后页面内任何操作都降不下内存，只有整页 reload 销毁模块作用域才能回收（实测 reload → 1.7 MB 吻合）。
