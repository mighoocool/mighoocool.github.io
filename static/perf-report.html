主线程渲染热点

按 self-time 排序 · 含调用链

下列函数即使主线程整体空闲，仍是播放期间反复执行、可以进一步压缩的部分。每张卡片左侧是采样重建的调用链（● 为叶子帧，↑ 向调用者回溯），右侧是原因与优化建议。

原因

播放时大量组件每帧重渲染，React 在 commit（mutation）阶段对每个变化的 DOM 节点走 updateProperties，逐个把 inline style 和 input 的 value 重新写回 DOM。时间轴、seek bar、数值显示这类每帧变化的组件，把大量成本花在了属性写入上。

优化建议

减少每帧重渲染的组件面积：对高频组件做 memo / 稳定 props。

绕开 React 逐帧改样式：把位置类样式改用 CSS transform，或对高频更新用 ref 命令式写入，避免走 updateProperties 的全量属性 diff+写。

原因

每个面板都被包在 React <Profiler> 里，每次 commit 都执行 perfInfo.current.innerText = ... —— 也就是面板左上角那个 528 / 0.1ms 计数器。多个面板 × 每帧刷新，是一直在付的调试开销。

优化建议

用 dev flag 或设置项门控这个性能覆盖层：关闭时不包 <Profiler>、不写 innerText。改动小、无风险，直接省掉每帧的 Profiler 回调与 DOM 写入。

明确的 Layout Thrashing

HoverTooltip 在 render body 里同步读布局

HoverTooltip.tsx:257
295 ms
调用链
●getBoundingClientRect
↑HoverTooltipHoverTooltip.tsx:257
↑renderWithHooksreact-dom
↑beginWorkreact-dom
原因

组件在渲染函数里直接调用 canvas.getBoundingClientRect()，还读了 offsetWidth / offsetHeight。3D 面板播放时持续重渲染，每次渲染都强制一次同步 layout —— 典型的 layout thrashing。

优化建议

用 ResizeObserver 缓存 canvas 边界，仅在尺寸真正变化时更新；把布局读取移出 render body（放到 useLayoutEffect 或事件回调），避免每帧强制回流。

原因

渲染循环里每帧把纹理（图像面板 / 3D 贴图）重新上传到 GPU。texSubImage2D 的开销随图像分辨率上升，图像面板活跃时才明显。

优化建议

仅在图像数据真正变化时置 texture.needsUpdate = true，避免每帧重传；可考虑 texStorage + 纹理复用，减少重新分配。
