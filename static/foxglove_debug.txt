  可优化的渲染热点（含调用链）

  A. React 提交阶段每帧重写 DOM 属性/样式 —— 最大块（合计 ~1.2s）

  set value        835ms ┐
  setValueForStyles 274ms ├─ commitMutationEffectsOnFiber
  set              123ms ┘    → updateProperties →
  updateWrapper/updateDOMProperties
  播放时大量组件每帧重渲染，React 在 commit 阶段把 inline style / input value
  逐个重新写进 DOM。
  优化方向：定位播放期间每帧重渲染的组件（时间轴/seek bar/数值显示），memo
  化；把高频变化的样式改用 CSS transform 或直接用 ref 命令式更新，避免走 React
  的 updateProperties 全量重写。

  B. 每个面板的 <Profiler> 调试计数器 —— onRender 115ms

  Panel.tsx:559 把每个面板都包在 React <Profiler> 里，每次 commit 写
  perfInfo.current.innerText（就是面板左上角那个 528 / 0.1ms
  计数）。这是生产环境一直在付的调试开销。
  优化方向：用 dev flag / 设置项门控这个性能覆盖层，默认关闭时不包
  <Profiler>、不写 innerText。

  C. HoverTooltip 在 render body 里同步读布局 —— getBoundingClientRect 295ms

  getBoundingClientRect ← HoverTooltip.tsx:257
  HoverTooltip.tsx:257 直接在渲染函数里 canvas.getBoundingClientRect() + 读
  offsetWidth/offsetHeight。3D 面板播放时持续重渲染 → 每次都强制同步
  layout（layout thrashing）。
  优化方向：用 ResizeObserver 缓存 canvas 边界，仅尺寸变化时更新；不要在 render
  body 里读布局。

  D. three.js 纹理上传 —— texSubImage2D 115ms

  WebGLRenderer.render → renderObjects → uploadTexture → texSubImage2D
  每帧上传纹理（图像面板 / 3D 贴图）。
  优化方向：仅在图像数据真正变化时置 texture.needsUpdate =
  true，避免每帧重传；可考虑 texStorage + 纹理复用。

  非渲染（worker 线程，仅供参考）

  #loadChunkData 563ms + lz4 解压 + parseChunk —— MCAP 数据加载/解压，运行在独立
  worker，与渲染解耦，目前不影响帧率。
